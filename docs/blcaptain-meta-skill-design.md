# Skill 能力产品化方法论设计文档

> 面向 Claude Skills、Codex Skills 与通用 Agent Skills 的全生命周期方法论设计稿  
> 目标：把人的专家经验、工作流、品味、工具调用和失败教训，转化成可调用、可分发、可验证、可迭代的 Agent 能力单元。

## 0. 执行摘要

Skill 不只是更长的提示词，也不是传统 App 的轻量替代品。它包含 prompt，但把 prompt 放进了更完整的工程化封装里：专家经验、流程判断、工具调用、模板资产、边界约束和真实失败案例，被组织成可被 Agent 按需加载、执行、验证和持续改进的能力单元。

本文档是 PRD 级方法论设计稿，不是最终可安装 Skill。它的职责是定义原则、结构、验收标准和后续实现路线；下一阶段才应把它拆成“薄 `SKILL.md` + 厚 `references/` + 可执行 `scripts/` + 可回归 `evals/` + 完整样例”的参照实现。

本方法论的核心判断是：

- Agent 不会天然抹平能力差距，它会放大目标、上下文、判断和系统组织能力的差距。
- Skill 的价值不是让模型“更会聊天”，而是让模型在具体任务里少走弯路、少犯重复错误、稳定交付结果。
- 好 Skill 的架构是“中心短，辐射厚”：入口文件短而高信号，重知识、脚本、模板和案例按需加载。
- 每个 Skill 都是一种上下文税；没有显著改变 Agent 行为的内容，都不应该进入 Skill。
- 真正稀缺的不是正向原则，而是真实失败后沉淀出的 gotchas、eval、边界和修复策略。
- 可发布 Skill 不是写完就结束，而是经历调研、分析、计划、开发、验证、测试、审计验收、发布、运营和迭代的能力资产。

一句话定义：

**Skill 是把可重复产生价值的专家行为，产品化为 Agent 可路由、可执行、可验证、可分发的能力包。**

## 1. 文档适用范围

### 1.1 目标读者

本文档面向多角色协作场景：

- AI 用户：想把重复任务、个人偏好和工作流沉淀成可复用能力。
- 产品经理：想把用户需求、使用路径和交付标准转化成 Agent 能力产品。
- 运营人员：想把 SOP、内容生产、用户触达和反馈闭环变成 Skill。
- 开发者与工程师：想把工具调用、脚本、测试、发布流程封装成可靠自动化能力。
- 测试人员：想为 Skill 建立触发、执行、回归和反例测试体系。
- 设计师：想把审美系统、版式规则、品牌资产和设计禁忌转成可执行约束。
- 创作者：想把文章、视频、PPT、图文卡片、品牌资产和分发流程变成内容生产飞轮。
- 行业专家：想把专业 SOP、判断标准和客户服务流程沉淀成可复用资产。

### 1.2 平台兼容边界

本文档不绑定单一 Agent 平台，默认兼容三类形态：

- Claude Skills：以 `SKILL.md` 为入口，配合 `references/`、`scripts/`、`assets/` 等资源。
- Codex Skills：以技能目录扩展 Codex 能力，强调触发描述、渐进披露、验证和工具协作。
- 通用 Agent Skills：任何能被 Agent 检索、加载、执行、验证和迭代的能力包。

不同平台的字段、安装路径、元数据格式和发现机制并不对称。Claude Skills 已经有较明确的 `SKILL.md` 与资源目录约定；Codex 当前更常见的稳定机制包括 `AGENTS.md`、项目规则、插件、工具和本地技能目录；通用 Agent Skills 更像跨平台设计原则集合。因此本文档只抽象共性原则：明确触发、低上下文成本、可执行流程、可验证结果、可维护资产。实现阶段必须重新核实目标平台的最新字段、路径、安装方式和验证工具。

### 1.3 平台差异速览

| 维度 | Claude Skills | Codex / OpenAI 工作流 | 通用 Agent Skills |
| --- | --- | --- | --- |
| 入口 | `SKILL.md` 与资源目录 | `AGENTS.md`、技能目录、插件、工具和项目上下文组合 | 任意可被 Agent 发现和加载的能力包 |
| 路由依据 | frontmatter、描述、上下文 | 项目规则、工具元数据、技能描述、用户请求 | name、description、manifest、索引或外部注册 |
| 资源组织 | `references/`、`scripts/`、`assets/` 等 | 取决于 Codex 环境和插件能力 | 平台自定义 |
| 验证重点 | 触发、执行、资源加载、forward-test | 项目规则遵守、工具调用、验证命令、用户改动保护 | 路由、执行、输出、权限、可移植 |
| 风险 | 误触发、上下文膨胀、脚本风险 | 项目上下文污染、工具权限、仓库改动风险 | 标准不一致、可移植性弱 |

PRD 要求：

- 不假设三类平台完全对称。
- 方法论写共性，具体实现写平台适配层。
- 第二轮实现可安装 Skill 时，必须为目标平台补单独的字段、路径和验证说明。

## 3. 第一性原理：Skill 的本质

### 3.1 Agent 能力公式

Agent 的实际产出不是由模型能力单独决定，而是由下面的系统共同决定：

```text
Agent 有效能力 =
模型能力
× 目标清晰度
× 上下文质量
× 工具可用性
× 流程约束
× 验证强度
× 反馈迭代速度
```

Skill 作用在后六项上。它不能替代模型本身，但能显著提高目标、上下文、工具、流程、验证和迭代的质量。

### 3.2 Skill 的启发式 ROI 模型

一个 Skill 是否值得做，取决于它是否产生正 ROI。下面不是可直接代入财务数值的严格公式，而是帮助判断方向的心智模型：

```text
Skill ROI =
（重复频次 × 单次任务价值 × 质量提升 × 分发系数）
-（构建成本 + 维护成本 + 上下文税 + 误触发成本 + 学习成本）
```

这意味着：

- 低频、低价值、一次性任务，不值得做成 Skill。
- 高频、标准化、容易出错、可验证、有明确交付标准的任务，适合做成 Skill。
- 如果 Skill 的 description 太宽，会产生误触发成本。
- 如果正文太长，会产生上下文税。
- 如果没有 eval，会产生质量波动成本。

落地时不要同时维护多套彼此独立的评分体系。建议这样使用：

- 早期机会判断：使用第 5.3 节的 24 分机会评分表。
- 发布前质量判断：使用第 21 章最终发布评分卡。
- 商业或团队投入判断：把本节 ROI 模型作为解释框架，补充定性原因和关键风险。

### 3.3 Skill 的关键单元模型

一个 Skill 不是一个文件，而是一条能力链路：

```text
用户真实需求
→ 触发表达
→ name / description 路由
→ SKILL.md 入口流程
→ 条件加载 references / scripts / assets
→ Agent 执行循环
→ 产物生成
→ 验证与修复
→ 失败进入 gotchas
→ 反馈进入下一版
```

其中最容易被低估的单元是：

- 触发表达：用户真实会怎么说。
- 反例表达：用户怎么说时不该加载。
- gotchas：真实失败后的专家边界。
- scripts：确定性逻辑下沉，减少模型自由发挥。
- assets：模板、骨架、字体、主题、示例等可复用资产。
- eval：证明 Skill 真的提升表现。

### 3.4 反直觉原则

1. **限制自由，才能提高下限。**  
   设计、写作、工程和运营 Skill 都不是让模型无限发挥，而是把高质量骨架、约束和检查机制交给模型。

2. **脚本通常比提示词更可靠。**  
   排序、解析、格式转换、构建、导出、校验等确定性任务，应优先交给脚本、CLI 或 API。

3. **description 不是广告，而是路由器。**  
   它的目标不是吸引人，而是让 Agent 在正确场景加载，并在错误场景不加载。

4. **gotchas 比原则更有价值。**  
   “要简洁”“要美观”“要测试”这类原则模型大多知道。真正有价值的是“不要连续三页同节奏”“文字不能压脸”“不要在无关任务加载本 Skill”。

5. **防盗的核心不是隐藏，而是复利。**  
   Skill 容易被仿，但持续迭代、案例、社区、品牌和分发会形成动态壁垒。

6. **Skill 与 prompt 不是二元对立。**  
   Skill 包含 prompt，但不止于 prompt。prompt 负责表达意图和行为边界；Skill 还要负责路由、资源、脚本、资产、验证、版本和分发。

7. **自动生成只能做初稿。**  
   真正可发布 Skill 需要人提供品味、失败样本、行业边界和业务取舍。

8. **人定义边界，Agent 维护长尾。**  
   理想状态不是完全手写，也不是完全自动沉淀。人定义品味、边界、责任和不可妥协的质量线；Agent 收集证据、提出改动、补 eval、维护长尾失败和生成候选迭代。

## 4. NABC：把 Skill 当成能力产品

### 4.1 Need：需求

用户的真实需求不是“我想要一个 Skill”，而是：

- 我有一类重复任务，每次都要重新解释。
- 我有一套经验，别人学不会或执行不稳定。
- 我有一个高频流程，靠聊天容易漏步骤。
- 我有一堆工具、模板和资料，Agent 不知道何时怎么用。
- 我想把自己或团队的专业能力变成可复用资产。

### 4.2 Approach：方案

用 Skill 能力产品化方法，把任务变成五层资产：

1. 路由层：name、description、触发和反触发。
2. 流程层：Agent 应该按什么顺序做事。
3. 约束层：哪些必须做，哪些禁止做，哪些需要询问。
4. 资源层：references、scripts、assets、模板、schema、示例。
5. 验证层：eval、gotchas、测试场景、审计清单。

### 4.3 Benefit：收益

对不同角色的收益不同：

| 角色 | 核心收益 |
| --- | --- |
| AI 用户 | 少写重复提示词，提升结果稳定性 |
| 产品 | 把能力封装成可理解、可使用、可运营的产品 |
| 运营 | 把 SOP 和内容飞轮标准化 |
| 开发 / 工程 | 把工具链、脚本、测试和发布流程沉淀 |
| 测试 | 建立触发、执行、回归和反例测试 |
| 设计师 | 把品味、版式、品牌规则变成可执行约束 |
| 创作者 | 把内容生产、视觉包装和分发流程变成复利系统 |
| 行业专家 | 把隐性经验变成可交付、可培训、可复制的资产 |

### 4.4 Competition：替代方案

| 替代方案 | 优点 | 局限 |
| --- | --- | --- |
| 普通 prompt | 低成本、快速 | 难版本化、难验证、易复制、难复用复杂资产 |
| README / SOP | 人类可读 | Agent 不一定知道何时加载，缺少路由和执行结构 |
| 单次对话 | 灵活 | 无法沉淀，不形成复利 |
| 自动生成 Skill | 快速出初稿 | 缺少真实失败、行业边界、品味和验证 |
| 大而全 Agent | 功能集中 | 上下文成本高、误用多、维护难 |

Skill 的优势在于：它把提示词、流程、脚本、模板、约束、失败经验和验证体系打包成可持续迭代的能力产品。

## 5. 什么值得做成 Skill

### 5.1 值得做的场景

满足越多，越值得做：

- 任务高频重复。
- 用户表达相对稳定。
- 有明确输入和输出。
- 结果好坏可判断。
- 过程容易漏步骤。
- 需要调用多个工具或文件。
- 有领域知识、审美、SOP 或业务规则。
- 有可复用模板、脚本、schema 或资产。
- 失败模式可被总结成 gotchas。
- 能通过案例、社区或组织内部传播复用。

### 5.2 不适合做的场景

- 一次性任务，未来几乎不会复用。
- 纯项目规则，应放在 `AGENTS.md`、`CLAUDE.md` 或项目说明中。
- 纯个人偏好，应放在 memory 或个人配置中。
- 已有成熟工具能直接解决，Skill 只是在包装常识。
- 没有明确交付物，也无法验证好坏。
- 触发边界过宽，容易污染其他任务。
- 维护成本高于复用收益。

### 5.3 Skill 机会评分表

每项 0-3 分，总分 24 分：

| 维度 | 0 分 | 1 分 | 2 分 | 3 分 |
| --- | --- | --- | --- | --- |
| 重复频次 | 几乎一次性 | 偶尔重复 | 经常重复 | 高频刚需 |
| 单次价值 | 可忽略 | 有帮助 | 明显节省时间 | 直接影响收入/质量 |
| 失败成本 | 低 | 中 | 高 | 很高且常见 |
| 专家经验密度 | 常识 | 少量经验 | 明显专业判断 | 高度依赖隐性经验 |
| 工具/资产复用 | 无 | 少量 | 多个稳定资源 | 复杂资源体系 |
| 可验证性 | 难判断 | 部分判断 | 多数可测 | 有明确 eval |
| 分发潜力 | 仅自用 | 小团队 | 社区可用 | 可成为能力商品 |
| 维护可控性 | 高成本 | 偏高 | 可控 | 易迭代 |

建议：

- 0-8 分：不要做 Skill，保留为 prompt 或笔记。
- 9-15 分：可做轻量 Skill 或项目规则。
- 16-20 分：值得做成正式 Skill。
- 21-24 分：值得产品化、发布和运营。

### 5.4 Non-Skill 决策树

专业的 Skill 方法论必须先学会拒绝 Skill 化。过度 Skill 化会制造维护负担、触发噪音、上下文税和用户理解成本。

```text
用户需求
├─ 只需要解释、总结、翻译、一次性回答？ → 直接回答
├─ 只需要一份可复用文字材料，但不需要 Agent 执行？ → 写文档 / 笔记 / SOP
├─ 任务主要是确定性处理，路由不是难点？ → 写脚本 / CLI / 自动化工具
├─ 是项目内约定、命令、测试方式或协作规则？ → 写 AGENTS.md / CLAUDE.md / 项目说明
├─ 是个人偏好、长期习惯或风格记忆？ → 写 memory / 用户偏好
└─ 高频复用、需要路由、边界、流程、资源和验证？ → 才做 Skill
```

判断原则：

- 能直接回答，就不要做成 Skill。
- 能写成普通文档，就不要伪装成 Skill。
- 能用一个脚本解决，就不要包一层复杂 Skill。
- 只有当“可发现、可路由、可复用、可验证、可维护”同时重要时，才进入 Skill 设计。

## 6. 方法论总流程

完整生命周期必须按以下顺序推进：

```text
阶段 0 真实交付验证 → 调研 → 分析 → 计划 → 开发 → 验证 → 测试 → 审计验收 → 发布 → 运营 → 迭代总结
```

每一步都有明确产物；没有产物，就等于没有完成。

### 6.1 阶段 0：真实交付验证

Skill 不应该从抽象概念开始。高质量 Skill 通常来自一次真实交付：先用 Agent 和人工判断完成一件真活，观察哪里反复解释、哪里容易失败、哪里依赖专家品味、哪里必须靠脚本或模板稳定下来。

进入正式 Skill 设计前，应至少满足以下条件之一：

- 已经完成过一次真实任务，并留下输入、过程、输出和修改记录。
- 已经观察到无 Skill 的 Agent 在具体步骤中失败。
- 已经有明确重复需求，且能拿出真实 query 或高保真模拟 query。
- 已经存在可复用模板、脚本、素材、SOP 或检查标准。

禁止行为：

- 只有一个想法，没有真实任务，就直接设计完整 Skill。
- 还没看到失败点，就先堆 references、scripts、evals 和治理文件。
- 把“未来可能用到”当成创建目录和配置项的理由。

阶段 0 产物：

```markdown
# 真实交付记录

## 任务背景
## 原始输入
## 交付过程
## 最终输出
## 用户修改意见
## Agent 失败点
## 可复用步骤
## 不值得 Skill 化的部分
```

### 6.2 轻量创作者快车道

多数个人创作者、设计师、运营和行业专家不需要一开始就进入治理体系。默认快车道如下：

```text
完成一次真实交付
→ 写 1 句 description
→ 写 20-60 行最小 SKILL.md
→ 准备 3 条触发测试：正例、反例、近邻例
→ 用 1 个真实输入跑一次
→ 发布给小范围用户
→ 把真实失败补成 gotchas
```

快车道只要求：

- 一个清晰触发场景。
- 一个最小工作流程。
- 一个明确输出标准。
- 三条路由测试。
- 一条真实 gotcha。

暂时不要求：

- 完整 governance。
- 多平台 packaging。
- regression history。
- 复杂 RACI。
- 大量 references。

当用户反馈、团队复用或路由混淆真实出现，再升级到 Production 或更高层级。

### 6.3 成熟度分层

不同 Skill 不应该使用同一套工程强度。方法论必须按复用范围、风险和维护要求选择最轻但足够可靠的形态。

| 层级 | 适用场景 | 默认产物 | 不该做 |
| --- | --- | --- | --- |
| Scaffold | 探索期、个人使用、短期流程 | 精简 `SKILL.md`，必要时少量 references | 不默认创建 scripts、evals、manifest、复杂报告 |
| Production | 团队复用、质量敏感、路由错误会浪费时间 | `SKILL.md`、接口元数据、必要 references、必要 scripts、基础 eval | 不做无关平台打包，不堆治理材料 |
| Library | 组织级复用、共享基础能力、需要跨环境维护 | 正反例 eval、near-neighbor、packaging expectations、维护元数据、可见回归证据 | 不让入口文件变重 |
| Governed | 关键业务、合规敏感、meta skill 或基础设施级能力 | owner、review cadence、maturity tier、lifecycle stage、regression history、promotion evidence | 不用治理外壳掩盖真实质量不足 |

升级规则：

- 默认停留在 Scaffold，直到复用真实发生。
- 团队复用或路由混淆明显时，升到 Production。
- 成为共享基础能力或跨平台能力时，升到 Library。
- 涉及关键业务、合规、组织级风险或 meta skill 时，升到 Governed。

### 6.4 最小可行严谨度

更高成熟度不等于更大的 `SKILL.md`。严谨度应该主要增长在 eval、reports、manifest、治理记录和脚本验证里，而不是增长在入口正文里。

```text
正确方向：更多证据 + 更硬门禁 + 更清晰边界 + 更少初始上下文
错误方向：更多口号 + 更多目录 + 更长正文 + 更多装饰性文件
```

## 7. 阶段一：调研

### 7.1 目标

充分收集真实任务、用户表达、现有流程、失败案例、竞品做法、工具资产和行业标准。

### 7.2 必问问题

- 用户真实想解决什么问题？
- 用户会用什么话触发这个需求？
- 这个任务现在怎么完成？
- 哪些步骤最容易漏？
- 哪些地方最依赖专家判断？
- 有哪些可复用模板、脚本、资料、案例？
- 失败时通常是什么样？
- 谁已经做过类似 Skill、插件、SOP 或工具？

### 7.3 调研材料

- 真实用户 query。
- 真实任务样本。
- 成功产物。
- 失败产物。
- 专家操作录屏或文字过程。
- 现有 prompt、SOP、模板、脚本。
- 开源 Skill 或类似工具。
- 用户评论、issue、反馈和改稿记录。

### 7.4 产物

形成《Skill 调研简报》：

```markdown
# Skill 调研简报

## 用户与场景
## 高频 query
## 输入材料
## 期望输出
## 当前流程
## 专家判断
## 可复用资产
## 失败案例
## 竞品 / 参考 Skill
## 初步机会评分
```

### 7.5 验收标准

- 至少有 5 条真实或高保真用户 query。
- 至少有 3 个成功案例和 3 个失败案例。
- 能说清楚“没有 Skill 时 Agent 会在哪里失败”。
- 能判断它是否值得做成 Skill，而不是只因为“想做”。

## 8. 阶段二：分析

### 8.1 目标

把调研材料转成明确的能力模型、业务价值、边界和资源结构。

### 8.2 分析框架

#### 第一性原理拆解

回答四个问题：

1. 这个任务的不可变目标是什么？
2. 哪些判断必须由人类专家定义？
3. 哪些步骤适合模型完成？
4. 哪些步骤必须交给脚本、CLI、API 或模板？

#### NABC 拆解

```markdown
Need：谁在什么场景下有什么痛点？
Approach：Skill 如何降低复杂度并提高交付稳定性？
Benefit：节省什么成本、提高什么结果、创造什么复利？
Competition：不用 Skill 时，用户会用什么替代方案？
```

#### 关键单元模型

拆出下面的单元：

- 触发单元：用户说什么时加载。
- 输入单元：需要什么材料。
- 处理单元：Agent 做哪些判断。
- 工具单元：脚本、CLI、API、MCP、浏览器、文件系统。
- 资产单元：模板、schema、素材、主题、案例。
- 输出单元：交付物是什么。
- 验证单元：如何判断合格。
- 失败单元：常见坑和禁止行为。
- 反馈单元：如何把失败写回 Skill。

### 8.3 角色视角分析

| 角色 | 关注点 | Skill 设计要回应 |
| --- | --- | --- |
| 用户 | 是否好用、结果是否靠谱 | 清楚输入、输出、使用路径和示例 |
| 产品 | 场景、门槛、体验、留存 | 明确定位、边界和反馈路径 |
| 运营 | 分发、案例、评价、增长 | 可展示结果、案例模板、传播素材 |
| 开发 | 文件结构、工具调用、依赖 | 资源分层、脚本、错误处理 |
| 测试 | 触发、执行、回归 | eval、正反例、跨模型测试 |
| 设计 | 审美、版式、品牌一致性 | 强约束、模板和后验检查 |
| 创作者 | 内容飞轮、复用、传播 | 产物可展示、可二次分发 |
| 行业专家 | 专业边界和责任 | 人机分工、禁用场景、专家复核 |

### 8.4 产物

形成《Skill 分析报告》：

```markdown
# Skill 分析报告

## 第一性原理
## NABC
## 用户 query 与触发边界
## 关键单元模型
## 人机分工
## 工具与资产地图
## ROI 测算
## 风险与禁区
```

### 8.5 验收标准

- 能明确说明“为什么是 Skill，而不是 prompt / README / App / MCP”。
- 能明确说明哪些内容进入 `SKILL.md`，哪些放入资源目录。
- 能明确说明不适用场景。
- 能用 ROI 解释是否值得维护。

### 8.6 品味萃取 Playbook

很多高价值 Skill 的核心不是流程，而是专家品味。品味不能只写成“要高级”“要专业”“要好看”，必须转成可检查约束。

萃取顺序：

1. 先问“什么样算错”，不要先问“什么样算好”。
2. 收集反面样本：丑的、乱的、廉价的、不合规的、不像行业内行做的。
3. 让专家逐条解释为什么错。
4. 把原因翻译成 gotchas、禁止项、检查项或模板约束。
5. 把高频错误转成 eval case。

访谈问题：

- 你看到什么结果会立刻觉得“不专业”？
- 哪些做法新手最爱用，但内行会避开？
- 哪些风格、词、颜色、动作、结构是禁区？
- 如果只能保留 5 条质量底线，你会保留什么？
- 哪些判断必须人工复核，不能交给 Agent 自由发挥？

转化示例：

| 专家表达 | 可执行约束 |
| --- | --- |
| “不要廉价感” | 禁止无意义紫色渐变、大面积发光、过度圆角和堆 emoji |
| “文字别压脸” | 图片有人脸时，标题不得覆盖五官；必要时加遮罩或换落点 |
| “别像模板套出来的” | 连续页面不得使用同一节奏和同一布局比例 |
| “法律意见不能乱给” | 高风险结论必须标记为需人工律师复核 |

好的品味萃取结果应进入：

- `SKILL.md` 的强约束。
- `references/examples.md` 的正反例。
- `evals/` 的失败回归。
- `assets/` 的模板和主题。

## 9. 阶段三：计划

### 9.1 目标

把分析结果转成可执行的 Skill 设计方案。

### 9.2 Skill 信息架构

推荐结构：

```text
skill-name/
├── SKILL.md
├── references/
│   ├── domain-knowledge.md
│   ├── examples.md
│   └── eval-cases.md
├── scripts/
│   └── deterministic-task.py
├── assets/
│   ├── templates/
│   ├── themes/
│   └── samples/
└── agents/
    └── openai.yaml 或平台元数据文件
```

不需要的目录不要创建。目录是为复用服务的，不是为了显得完整。

### 9.3 中心短，辐射厚

`SKILL.md` 应只包含：

- 触发后必须知道的核心原则。
- 关键流程。
- 资源导航。
- 强约束。
- gotchas。
- 验证步骤。

`references/` 应包含：

- 长文档。
- 领域知识。
- API 说明。
- 案例库。
- 评估样本。

`scripts/` 应包含：

- 确定性处理。
- 可重复执行的转换、校验、导出、批处理。
- 容易被模型写错的逻辑。

`assets/` 应包含：

- 模板。
- 字体。
- 图片。
- 主题。
- schema。
- 示例输入和输出。

### 9.4 description 设计

description 是路由触发器。写法：

```yaml
description: Use when [用户任务场景] for [具体输入/输出/对象], especially when [典型触发条件].
```

中文设计时也要遵循同一原则：

```yaml
description: 当用户需要把文章、大纲或演讲内容转成可演示幻灯片，并要求版式、节奏、视觉一致性和导出质量时使用。
```

不要写：

```yaml
description: 一个能生成高级漂亮 PPT 的超强 Skill。
```

### 9.4.1 路由语言策略

description 的语言会影响触发。中文用户、英文用户和双语团队的 query 分布不同，不能只凭作者习惯写路由语言。

建议：

- 面向中文用户：description 可用中文，但应覆盖用户真实中文表达。
- 面向英文用户：description 优先英文，并用英文 query 做正反例。
- 面向双语用户：description 可用主语言 + 关键英文同义词，或在 eval 中显式覆盖双语触发。
- 面向跨平台公开发布：优先使用目标平台最容易检索的主语言，同时在 README / 发布页补多语言说明。

测试要求：

- 至少准备 3 条主语言正例。
- 如果面向双语场景，至少准备 2 条第二语言正例和 2 条近邻反例。
- 如果 description 改语言，必须重跑 route eval。

### 9.5 计划产物

形成《Skill 设计方案》：

```markdown
# Skill 设计方案

## 名称
## description
## 目标用户
## 适用场景
## 不适用场景
## 输入
## 输出
## 流程
## 文件结构
## scripts
## references
## assets
## gotchas
## eval 设计
## 发布计划
## 成熟度层级
## 风险门禁
```

### 9.6 验收标准

- 文件结构服务于任务，不为完整而完整。
- description 有正触发和反触发考虑。
- 每个资源文件都有加载条件。
- 每个流程步骤都有验证方式。

### 9.7 风险门禁计划

计划阶段要先决定需要哪些质量门禁，而不是开发后再临时补测试。

| 风险等级 | 常见层级 | 最小门禁 |
| --- | --- | --- |
| 低 | Scaffold | 结构校验、资源边界检查、人工样例审阅 |
| 中 | Production | 低风险门禁 + trigger eval + 正反例 |
| 高 | Library | 中风险门禁 + description optimization、near-neighbor、route confusion、packaging validation |
| 关键 | Governed | 高风险门禁 + governance scoring、regression history、promotion policy、review cadence |

门禁选择原则：

- 路由可能混淆时，优先加 trigger gate。
- 多客户端分发时，优先加 packaging gate。
- 团队或组织复用时，优先加 governance gate。
- 任何非琐碎 Skill 都应有 context gate。
- 新增门禁如果只增加复杂度，不提升可靠性，应删除或降级。

### 9.8 Skill 拆分与组合

高级 Skill 往往不是单体，而是组合系统。拆分的目的不是显得架构复杂，而是让每个能力边界清晰、可测试、可替换。

应该拆分的信号：

- 一个 Skill 同时服务多个明显不同的用户目标。
- 一个流程中既有创作、又有导出、又有发布、又有数据同步，失败模式完全不同。
- 某个子流程可被多个 Skill 复用。
- 某部分需要脚本或权限，另一部分只需要文本判断。
- 不同平台或模型能力差异导致执行路径需要分层。

不该拆分的信号：

- 只是为了目录好看。
- 子 Skill 之间高度耦合，单独不可理解。
- 拆分后用户更难知道该调用谁。
- 当前还没有真实复用需求。

组合设计模板：

```markdown
# Skill 组合设计

## 父 Skill 负责什么
## 子 Skill 列表
## 每个子 Skill 的输入 / 输出
## 依赖关系
## 版本联动
## 失败时如何降级
## 哪些子能力可以独立发布
```

例：

- Logo Skill 可拆成 Logo 本体生成、展示图生成、WebGL 背景生成。
- 内容分发 Skill 可拆成内容提炼、图文卡片排版、平台发布检查。
- 游戏原型 Skill 可拆成玩法设计、素材处理、视频生成、打包发布。

## 10. 阶段四：开发

### 10.1 目标

把设计方案实现为可读、可调用、可维护的 Skill 或方法论文档。

### 10.2 `SKILL.md` 写作原则

- 用命令式语言告诉 Agent 怎么做。
- 不写显而易见的常识。
- 不写宣传文案。
- 不写人类阅读的长篇背景故事。
- 保留能改变 Agent 行为的判断。
- 把长内容移入 references。
- 把确定性步骤移入 scripts。
- 把模板和样例移入 assets。

### 10.3 推荐 `SKILL.md` 结构

```markdown
---
name: skill-name
description: Use when ...
---

# Skill Name

## 核心原则
## 工作流程
## 资源选择
## 质量标准
## Gotchas
## 验证
```

### 10.4 开发顺序

1. 先写 eval 场景草案。
2. 再写 description。
3. 再写最小 `SKILL.md`。
4. 再补 references、scripts、assets。
5. 再根据测试补 gotchas。
6. 最后删掉不会改变行为的内容。

这个顺序的原因是：如果先写正文，容易把自己的表达欲当成用户价值；如果先写 eval，会被迫回到真实任务。

### 10.5 开发验收

- `SKILL.md` 能在 3 分钟内被读懂。
- 文件结构没有无用目录。
- 每个脚本可以独立运行。
- 每个引用文件都有明确加载条件。
- 每个 gotcha 来自真实或高保真失败场景。

## 11. 阶段五：验证

### 11.1 目标

证明 Skill 的结构、路由、资源和基本流程没有明显错误。

### 11.2 验证类型

| 类型 | 检查内容 |
| --- | --- |
| 结构验证 | 文件名、frontmatter、目录结构是否符合平台要求 |
| 路由验证 | 应加载时是否加载，不应加载时是否不加载 |
| 资源验证 | references、scripts、assets 是否可被找到并正确使用 |
| 流程验证 | Agent 是否按预期步骤执行 |
| 输出验证 | 产物是否符合格式和质量标准 |

### 11.3 description 验证

构建三类 query：

```markdown
## 正例
用户这样说时必须触发 Skill。

## 边界例
用户这样说时可能触发，需要根据上下文判断。

## 反例
用户这样说时禁止触发 Skill。
```

例：

```markdown
正例：帮我把这篇文章做成 12 页演示用 HTML PPT。
边界例：帮我整理这篇文章的大纲。
反例：帮我解释 HTML 的 section 标签。
```

### 11.4 验证产物

形成《Skill 验证记录》：

```markdown
# Skill 验证记录

## 结构验证
## description 正例
## description 反例
## 资源加载验证
## 流程验证
## 输出验证
## 发现的问题
## 修复记录
```

### 11.5 上下文预算与资源边界验证

上下文纪律需要量化，但不应伪精确。不同平台 tokenizer、系统提示和会话上下文不同，下面的数值只能作为初始参考阈值；实现时必须用目标平台真实 tokenizer 或上下文估算工具实测。

| 层级 | 初始参考阈值 | 设计含义 |
| --- | ---: | --- |
| Scaffold | 700 tokens 左右 | 只保留触发、核心流程和输出契约 |
| Production | 1000 tokens 左右 | 允许少量质量门禁与资源导航 |
| Library | 1300 tokens 左右 | 允许维护和可移植性提示，但正文仍要克制 |
| Governed | 1300 tokens 左右 | 治理证据应放在 manifest / reports，而不是塞进入口 |

校准口径：

- 用目标平台 tokenizer 实测，而不是跨平台复用单一数字。
- 同时记录 token 数和“占典型可用上下文比例”。
- 若 skill 将被频繁触发，预算应更严。
- 若 skill 只在长程任务中按需触发，可把重内容放入 references，并保持入口轻。

资源边界检查：

- `SKILL.md` 是否只包含触发、核心骨架、输出契约、安全默认值和分支选择？
- `references/` 是否被 `SKILL.md` 明确引用，且有加载条件？
- `scripts/` 是否解决真实确定性问题，而不是装饰性工程化？
- `evals/` 是否对应真实路由风险、质量声明或复用压力？
- `assets/` 是否承载输出所需静态资源，而不是把样例堆进上下文？
- 是否存在未被引用、未被声明、无人使用的装饰性目录？

质量密度判断：

```text
质量密度 = 可靠性证据 / 初始上下文成本
```

好的 Skill 不是文件最多，而是在尽可能小的初始加载里，保留足够强的边界、执行和验证证据。

## 12. 阶段六：测试

### 12.1 目标

证明 Skill 在真实或高保真任务中确实提高结果，而不是只在文档层面看起来合理。

### 12.2 测试矩阵

| 测试类型 | 目的 | 示例 |
| --- | --- | --- |
| 基线测试 | 无 Skill 时 Agent 会怎么失败 | 直接让 Agent 做任务，记录漏步骤和错误 |
| 正例测试 | Skill 是否能完成目标任务 | 用典型 query 跑完整流程 |
| 反例测试 | Skill 是否会误加载 | 用相似但无关 query 测试 |
| 压力测试 | 在复杂输入下是否稳定 | 长文、脏数据、多约束、时间压力 |
| 跨模型测试 | 不同模型是否都能执行 | 强模型看上限，弱模型保下限 |
| 回归测试 | 修改后是否破坏旧能力 | 旧案例重新跑 |

### 12.3 可执行 Eval 协议

Eval 不只是“有几条测试”。它必须说明在哪里测、怎么测、如何判定通过。

最低用例数：

| 层级 | 最低用例 |
| --- | --- |
| Scaffold | 1 个真实输入 + 3 条路由用例 |
| Production | 3-5 个执行用例 + 正例/反例/近邻路由 |
| Library | 5+ 个执行用例 + train/dev/holdout 或等价分组 |
| Governed | Library 基础上增加回归历史、失败库、晋升策略 |

路由测试必须分开三类：

- 正例：应该触发该 Skill。
- 反例：不应该触发任何 Skill 或应该直接回答。
- 近邻例：看起来相似，但应该触发另一个 Skill 或另一个流程。

关键操作要求：

- forbidden-load 测试必须使用 fresh context。不能在当前已经读过 Skill 的会话里问“你会不会加载它”，那是失真的。
- near-neighbor 测试必须在多 Skill 环境下进行。单个 Skill 独立测试暴露不出路由污染。
- 基线测试必须记录无 Skill 时的失败，而不是只记录有 Skill 后的成功。
- 修改 description 后必须重跑旧反例，防止扩大触发面。
- 修改正文后必须重跑旧执行用例，防止破坏已验证流程。

评分口径：

```markdown
# Eval Result

## 用例 ID
## 类型：正例 / 反例 / 近邻 / 执行 / 压力 / 回归
## Fresh context：是 / 否
## 多 Skill 环境：是 / 否
## 期望行为
## 实际行为
## 通过：是 / 否
## 失败原因
## 是否需要改 description
## 是否需要改 SKILL.md
## 是否需要新增 gotcha
```

通过标准：

- 正例召回稳定。
- 反例不误触发。
- 近邻不污染。
- 执行用例能产出合格交付物。
- 旧回归不被新改动破坏。

### 12.4 三个标准测试场景

#### 场景 A：内容创作者做图文 Skill

输入：一篇长文章、一组截图、目标平台。  
期望：输出适配平台的图文卡片方案和视觉产物。  
重点测试：版式选择、断行、图片裁切、平台风格、禁用廉价视觉套路。

#### 场景 B：企业运营做 SOP Skill

输入：客服流程、用户分层、话术、CRM 字段。  
期望：输出标准处理流程、分支判断和记录格式。  
重点测试：边界判断、敏感场景升级、字段完整性、合规表达。

#### 场景 C：工程师做开发流程 Skill

输入：功能需求、代码仓库、测试命令。  
期望：输出计划、实现、测试、审查和发布步骤。  
重点测试：是否先读项目规则、是否限制改动范围、是否真实运行验证。

### 12.5 测试通过标准

- 相比无 Skill 基线，步骤遗漏明显减少。
- 输出质量更稳定。
- 反例不误触发。
- 失败能被记录为 gotcha 或 eval。
- 修改后能通过旧回归案例。

### 12.6 失败库与反模式回归

gotchas 不应只停留在 prose。高价值失败应进入 failure library，并尽可能转成可回归案例。

失败库至少包含：

| 失败族群 | 典型表现 | 回归目标 |
| --- | --- | --- |
| one-off-vs-reusable | 一次性问题被包装成 Skill | 防止过度 Skill 化 |
| explain-not-package | 用户只是想理解概念，却被引导建包 | 防止错误升级 |
| document-export-vs-agent-skill | 用户要一份文档，不需要 Agent 执行能力 | 防止把文档误做成 Skill |
| future-outline-vs-build | 用户只是设想未来方向，却被直接开发 | 防止过早实现 |
| route-near-neighbor | 相似任务误触发本 Skill | 防止边界污染 |
| context-bloat | 正文越来越长，资源分层失效 | 防止上下文失控 |

失败记录模板：

```markdown
# Failure Case

## 失败族群
## 用户请求
## 错误行为
## 正确行为
## 应触发 / 不应触发
## 需要新增或修改的 eval
## 是否需要修改 description
```

高成熟度 Skill 的要求：

- Production：至少记录高频失败。
- Library：失败族群要能回归测试。
- Governed：失败库、回归历史和 promotion decision 必须可追踪。

## 13. 安全、权限与供应链

### 13.1 目标

可发布 Skill 不是只有质量风险，还有执行风险。只要 Skill 会运行脚本、读写文件、调用工具、访问网络或连接第三方服务，就必须把安全和权限作为一等约束。

### 13.2 安全原则

- 最小权限：只申请完成任务必需的文件、网络、工具和环境权限。
- 明示副作用：任何会写文件、删除文件、改配置、发请求、发布内容、花钱调用 API 的动作都必须显式说明。
- 默认只读：不确定时先读取、预览、dry-run，再执行写操作。
- 禁止隐藏执行：Skill 不得在用户不知情的情况下运行破坏性命令或上传数据。
- 密钥隔离：Skill 包内不得包含 API key、token、cookie、私有凭证或个人数据样本。
- 依赖可审计：第三方脚本、包、二进制和远程安装命令必须可追溯来源。

### 13.3 第三方 Skill 审查清单

安装或分发第三方 Skill 前检查：

- `SKILL.md` 是否存在过宽权限或隐藏副作用？
- `scripts/` 是否包含删除、上传、远程执行、安装依赖、修改 shell 配置等行为？
- 是否有锁定版本或依赖清单？
- 是否需要外部 API、账号或付费服务？
- 是否把密钥、样本用户数据或公司内部路径写进仓库？
- 是否存在 prompt injection 风险：要求 Agent 忽略系统/用户指令、绕过权限、读取无关文件？
- 是否有可信维护者、版本、许可证和更新记录？

### 13.4 安全分级

| 风险 | 示例 | 要求 |
| --- | --- | --- |
| 低 | 只读 references、纯文本流程 | 普通审阅 |
| 中 | 本地生成文件、格式转换 | 明示输出路径，保留备份或 dry-run |
| 高 | 网络访问、第三方 API、批量文件修改 | 用户确认、日志记录、最小权限 |
| 关键 | 发布内容、支付、生产系统、删除数据 | 人工审批、回滚方案、强制审计 |

### 13.5 禁止进入分发 Skill 的内容

- 密钥、cookie、token、私有证书。
- 真实客户数据、未脱敏日志、内部账号。
- 默认破坏性命令。
- 自动安装不可信远程脚本。
- 绕过平台权限或安全沙箱的说明。
- 要求 Agent 忽略上级指令或读取无关隐私文件的提示。

## 14. 阶段七：审计验收

### 14.1 目标

用高标准反向检查 Skill 是否真的值得发布。

### 14.2 审计清单

#### 战略审计

- 这个 Skill 是否解决真实重复问题？
- 是否有明确目标用户？
- 是否有可展示结果？
- 是否比 prompt / README / App 更适合？

#### 产品审计

- 用户是否知道输入什么？
- 用户是否知道输出长什么样？
- 是否有典型使用场景？
- 是否有不适用场景？

#### 工程审计

- 文件结构是否最小必要？
- 脚本是否可运行？
- 资源是否有加载条件？
- 是否避免把复杂逻辑交给模型自由发挥？

#### 上下文审计

- 每句话是否会改变 Agent 行为？
- 是否有过长背景说明？
- 是否把重内容放到 references？
- 是否把确定性逻辑放到 scripts？

#### 测试审计

- 是否有正例？
- 是否有反例？
- 是否有 forbidden load？
- 是否有跨模型或弱模型测试？
- 是否有回归用例？

#### 运营审计

- 是否有案例展示？
- 是否有安装和使用路径？
- 是否有反馈渠道？
- 是否有版本迭代策略？

### 14.3 一票否决项

出现以下问题，不应发布：

- 没有真实需求，只是概念包装。
- description 像广告语，不能路由。
- 正文大段讲常识，没有改变 Agent 行为。
- 没有 eval 或反例。
- 没有 gotchas。
- 没有明确输入输出。
- 工具和资源无法运行或无法找到。
- 误触发成本很高。
- 用户无法理解它解决什么问题。

### 14.4 治理元数据

当 Skill 从个人探索进入团队复用后，必须被当成可治理资产，而不是一次性 prompt 文件。

可复用或更高层级 Skill 应声明：

```json
{
  "name": "skill-name",
  "version": "0.1.0",
  "owner": "person-or-team",
  "updated_at": "YYYY-MM-DD",
  "review_cadence": "monthly | quarterly | semiannual | annual | per-release",
  "status": "experimental | active | deprecated",
  "maturity_tier": "scaffold | production | library | governed",
  "lifecycle_stage": "scaffold | production | library | governed"
}
```

治理审计问题：

- 是否有明确 owner？
- 是否有 review cadence？
- 声称的成熟度是否匹配实际 eval、资源和维护证据？
- deprecated Skill 是否有替代方案或迁移说明？
- 是否有 regression history 或失败记录证明它没有漂移？

治理动作：

- 保持现状。
- 收紧触发边界。
- 拆分成兄弟 Skill。
- 把正文细节迁移到 references。
- 把脆弱逻辑迁移到 scripts。
- 降级、弃用或替换。

## 15. 阶段八：发布

### 15.1 发布目标

让目标用户能理解、安装、调用、验证和传播 Skill。

### 15.2 发布资产

| 资产 | 目的 |
| --- | --- |
| Skill 目录 | 可安装、可运行 |
| README 或发布页 | 给人类用户理解，不一定进入 Skill 目录 |
| 示例输入 | 降低使用门槛 |
| 示例输出 | 展示结果 |
| 截图 / 视频 | 提升转化 |
| 安装说明 | 降低使用阻力 |
| 更新日志 | 说明版本变化 |
| FAQ / gotchas | 降低失败率 |

注意：有些平台不建议在 Skill 目录里放 README、安装指南等冗余文件。可以把人类发布页放在仓库、网站或社区页面中，而不是塞进 Agent 加载目录。

### 15.3 分发策略

- GitHub：适合代码型 Skill、版本管理、开源协作、跨平台覆盖。
- 平台 Skillhub / 应用商店：适合低门槛安装、推荐和商业转化。
- 小红书 / Twitter / 公众号：适合展示结果、讲案例、做种草和反馈收集。
- 视频平台：适合演示复杂工作流。
- 社群 / 企业内部：适合行业 SOP 和组织知识沉淀。

### 15.4 发布前检查

- 用户能在 30 秒内理解它解决什么问题。
- 用户能在 3 分钟内跑第一个例子。
- 有至少一个高质量结果展示。
- 有清晰安装方式。
- 有明确反馈渠道。

## 16. 阶段九：运营

### 16.1 目标

让 Skill 从一次发布变成持续复利资产。

### 16.2 运营飞轮

```text
真实任务
→ 高质量交付
→ 沉淀 Skill
→ 产出案例
→ 内容分发
→ 用户反馈
→ gotchas / eval / 模板更新
→ 新版本
→ 更多真实任务
```

### 16.3 反馈筛选

不是所有反馈都要吸收。只吸收三类：

- 能提升泛化能力的反馈。
- 能减少高频失败的反馈。
- 能明确改善目标用户体验的反馈。

不要吸收：

- 单个用户极端偏好。
- 与定位冲突的需求。
- 会显著增加上下文税但收益很低的内容。
- 把 Skill 拉向大而全的功能膨胀。

### 16.4 运营指标

运营指标要区分“创作者能观察到的信号”和“平台/组织才有的遥测”。不要把拿不到的数据写成个人创作者的待办。

创作者可观测指标：

| 指标 | 说明 |
| --- | --- |
| 自己复用次数 | 作者是否在真实工作中继续使用 |
| 首次样例成功率 | 用户照着 README / 示例是否能跑出结果 |
| issue / 评论 | 用户在哪里失败、哪里看不懂 |
| star / fork / 收藏 / 转发 | 分发与兴趣信号，不等同于质量 |
| 用户案例投稿 | 是否有人愿意展示产物 |
| 模板 / PR 贡献 | 是否出现社区共建 |
| 版本反馈闭环 | 新版是否解决旧评论和旧 issue |

平台或组织级遥测指标：

| 指标 | 说明 |
| --- | --- |
| 激活率 | 安装后是否真的使用 |
| 首次成功率 | 第一次使用是否产出合格结果 |
| 复用率 | 用户是否反复调用 |
| 失败率 | 任务失败或需要大量人工修复的比例 |
| 误触发率 | 不该加载时加载的比例 |
| 留存率 | 用户是否持续使用 |

使用原则：

- 独立发布时，不要伪造平台级指标。
- GitHub 和社交平台指标只能代表兴趣和传播，不能直接证明 Skill 质量。
- 真正的质量证据仍来自 eval、真实案例、失败修复和版本回归。

## 17. 阶段十：迭代总结

### 17.1 目标

让每一次真实使用都转化为下一版 Skill 的证据。

### 17.2 迭代节奏

小版本：

- 修 description。
- 补 gotchas。
- 修脚本 bug。
- 补反例。

中版本：

- 调整流程。
- 新增模板。
- 拆分 references。
- 增加测试场景。

大版本：

- 重定义目标用户。
- 拆分成多个 Skill。
- 合并重复 Skill。
- 改变技术路线。

### 17.3 迭代记录模板

```markdown
# Skill 迭代记录

## 版本
## 触发原因
## 用户反馈 / 失败案例
## 修改内容
## 新增 eval
## 新增 gotchas
## 回归测试结果
## 残余风险
```

## 18. 多角色协作模型

### 18.1 RACI

| 工作项 | 负责人 | 参与者 | 咨询者 | 知情者 |
| --- | --- | --- | --- | --- |
| 需求定义 | 产品 | 用户、运营 | 行业专家 | 团队 |
| 专家经验抽取 | 行业专家 | 产品、创作者 | 测试 | 团队 |

| 资源建设 | 开发 / 设计 | 创作者 | 行业专家 | 产品 |

### 18.2 交接物

- 产品给工程：用户、场景、输入输出、边界。
- 专家给产品：判断标准、禁忌、失败案例。
- 设计给工程：模板、主题、视觉约束、检查标准。
- 工程给测试：文件结构、脚本、依赖、运行方式。
- 测试给全员：eval、反例、失败报告。
- 运营给产品：案例、评论、传播数据、用户语言。

## 20. 典型案例推演

### 20.1 内容创作者：图文卡片 Skill

需求：把长文章、观点或案例转成适合社交平台传播的图文卡片。  
关键经验：平台比例、信息密度、断行、截图美化、真实图片优先、AI 生图兜底。  
适合资源：版式骨架、主题色板、截图模板、示例输出。  
gotchas：文字压脸、连续卡片节奏重复、廉价渐变、平台风格错位。  
eval：同一文章生成 3 套卡片，检查可读性、平台适配、视觉一致性。

### 20.2 企业运营：客户跟进 SOP Skill

需求：把客户分层、跟进节奏、话术、CRM 记录标准转成 Agent 辅助流程。  
关键经验：客户状态判断、升级条件、禁用话术、合规边界。  
适合资源：话术库、字段 schema、行业政策、CRM 示例。  
gotchas：承诺过度、漏记字段、未升级高风险客户、机械套话。  
eval：模拟不同客户状态，检查分支判断和记录完整性。

### 20.3 工程师：功能开发流程 Skill

需求：让 Agent 按团队标准完成需求分析、计划、实现、测试和审查。  
关键经验：先读项目规则、限制改动范围、先写测试、真实运行验证。  
适合资源：项目命令、测试模板、PR checklist、代码风格。  
gotchas：顺手重构、没跑测试、只口头验证、忽略用户已有改动。  
eval：用小功能和 bugfix 场景测试是否遵守流程。

### 20.4 填满样例：社交媒体图文卡片 Skill

#### 真实交付记录

任务背景：把一篇 3000 字观点文章转成 6 张 3:4 竖版图文卡片，用于小红书和公众号首图分发。  
原始输入：文章正文、作者头像、2 张产品截图、目标平台“小红书优先，公众号可复用”。  
最终输出：6 张竖版卡片，每张一个核心观点，首张做强标题，后 5 张做递进论证。  
Agent 失败点：初版使用了紫色渐变、标题过长、截图裁切掉关键信息、连续 4 张同版式、文字覆盖人物脸部。  
可复用步骤：提炼观点、拆卡片节奏、选择版式骨架、处理截图、做可读性检查、导出平台尺寸。

#### Skill Brief

```markdown
# Skill Brief

## 名称
social-card-maker

## 一句话定义
把文章、观点或案例转成适合社交平台传播的竖版图文卡片。

## 目标用户
内容创作者、产品运营、独立开发者、活动策划。

## 高频任务
文章转卡片、发布会观点转卡片、产品更新转卡片、案例复盘转卡片。

## 输入
文章正文、平台、图片/截图、品牌色或参考风格。

## 输出
3:4 竖版卡片方案、每张卡片文案、版式选择、素材使用说明、导出检查清单。

## 不适用场景
用户只想摘要文章；用户只要普通长图；用户需要自动发布到平台。
```

```markdown
---
name: social-card-maker
description: 当用户需要把文章、观点、案例或产品更新转成适合小红书、公众号、Twitter 等平台传播的竖版图文卡片，并要求版式、断行、截图、真实图片和平台风格适配时使用。
---

# Social Card Maker

## 核心原则
- 先确认平台、卡片数量、素材来源和输出格式。
- 优先真实图片、截图和产品素材；AI 生图只作兜底。
- 先提炼观点，再选择版式，不从视觉模板反推内容。

## 流程
1. 读取文章，提炼 3-8 个可独立传播的观点。
2. 为每张卡片分配角色：封面、问题、洞察、证据、方法、收束。
3. 选择版式骨架，避免连续三张同节奏。
4. 处理图片和截图，确保主体、文字和可读区域不冲突。
5. 输出每张卡片的标题、正文、素材、版式和检查项。

## Gotchas
- 不使用无意义紫色渐变、发光和大面积 blur 作为主视觉捷径。
- 文字不得覆盖人物五官、产品关键 UI 或截图核心数据。
- 单张卡片只讲一个核心观点。
- 连续卡片不得使用完全相同的信息节奏。

## 验证
- 检查 1 秒内能否读懂首张卡片。
- 检查每张卡片是否有明确主次层级。
- 检查平台比例、断行、图片裁切和文字安全区。
```

#### Gotchas

| gotcha | 正确处理 |
| --- | --- |
| 标题贪长 | 每张只保留一个传播钩子，副信息下沉 |
| 连续同版式 | 封面、观点、证据、方法、总结交替 |
| 截图不可读 | 放大关键区域，裁掉无关 chrome，补标注 |
| 文字压脸 | 换落点、加遮罩、换图或降低文字密度 |
| AI 图抢戏 | 没有真实素材时才使用，且不得替代产品证据 |

## 21. 最终发布评分卡

每项 0-2 分，总分 34 分：

| 维度 | 检查问题 |
| --- | --- |
| 真实需求 | 是否来自真实重复任务 |
| 用户表达 | 是否有真实 query |
| 触发准确 | description 是否能正确路由 |
| 边界清楚 | 是否写明不适用场景 |
| 流程可执行 | Agent 是否知道下一步做什么 |
| 资源分层 | 是否中心短、辐射厚 |
| 脚本可靠 | 确定性任务是否下沉 |
| 模板资产 | 是否有可复用骨架 |
| gotchas | 是否来自真实失败 |
| eval | 是否有正例、反例、压力例 |
| 输出质量 | 是否有明确验收标准 |
| 上下文成本 | 是否删除无效内容 |
| 运营资产 | 是否有案例和发布页 |
| 迭代机制 | 是否有反馈进入下一版 |
| ROI | 是否收益大于维护成本 |
| 成熟度匹配 | Scaffold / Production / Library / Governed 是否与实际复用风险匹配 |
| 治理元数据 | 团队级以上 Skill 是否有 owner、version、review cadence 和 lifecycle |

建议：

- 0-17 分：不要发布。
- 18-25 分：内部试用。
- 26-30 分：可公开发布。
- 31-34 分：可作为标杆 Skill 运营。

## 22. 结论

Skill 的本质，是把人的可重复高价值行为组织成能力资产。

它不是 prompt 工程的终点，而是 Agent 时代的产品工程、上下文工程、知识工程、测试工程和运营工程的交汇点。

一个真正好的 Skill，必须同时满足五个条件：

1. 有真实需求，不是概念包装。
2. 有专家判断，不是常识堆叠。
3. 有工程结构，不是长提示词。
4. 有验证体系，不是自我感觉良好。
5. 有运营迭代，不是一次性发布。

如果把每一次真实任务都当成能力资产的原料，把每一次失败都变成 gotcha，把每一次成功都变成案例，把每一次反馈都变成 eval，那么 Skill 就不只是一个工具，而会成为个人、团队和组织在 Agent 时代的复利系统。

| 缺口 | 为什么还存在 | 下一步 |
| --- | --- | --- |
| 还不是可安装 Skill | 当前用户明确要求先做 PRD | 第二轮拆成 `SKILL.md`、`references/`、`assets/`、`evals/`、`scripts/` |

| 未做真实 token 计量 | 跨平台 tokenizer 不一致 | 实现时用目标平台 tokenizer 和真实上下文测量 |
| 未做真实平台适配 | Claude、Codex、通用 Agent Skills 机制不对称 | 第二轮按目标平台分别写 adapter 和安装说明 |

### 23.3 设计层压力测试

本文档已做过三个设计层推演：内容创作者图文 Skill、企业运营 SOP Skill、工程开发流程 Skill。结论是：PRD 能指导三类场景推导出 Skill 方案，但这些只是设计层推演，不等同于真实 Agent 执行测试。

第二轮实现时，三类场景都必须补：

- 一个真实输入。
- 一个真实输出。
- 三条路由用例。
- 一条 near-neighbor。
- 一条失败案例。
- 一条修复后的 gotcha。

### 23.4 残余风险

- 文档仍然较长，不应作为最终 `SKILL.md` 加载。
- 部分章节仍有重叠，第二轮拆分 references 时需要进一步去重。
- 安全章节是方法论级审查清单，不替代真实代码审计。
- 平台差异仍需实现阶段复核最新官方规范。
- 评分卡仍是决策辅助，不应被当成客观质量证明。
