# 真实交付优先

## 加载时机

当用户还没有清晰样本、只给想法、要求从零做 Skill、或需要判断“是否过早产品化”时加载。

## 核心原则

高质量 Skill 来自真实交付，不来自抽象愿望。先完成一次真活，记录 Agent 自然失败点，再把可复用部分产品化。

## 阶段 0 证据

至少满足一项：

- 已完成真实任务，并留下输入、过程、输出和修改记录。
- 已观察到无 Skill 的 Agent 在具体步骤中失败。
- 已有真实 query 或高保真模拟 query。
- 已有可复用模板、脚本、素材、SOP 或检查标准。

## 禁止信号

- 只有“想做一个很强 Skill”的概念，没有真实任务。
- 没有看到失败点就堆 `references/`、`scripts/`、`evals/`。
- 用治理文件替代真实样例、真实输出和真实验证。

## 真实交付记录模板

```markdown
# 真实交付记录

## 任务背景
## 原始输入
## 交付过程
## 最终输出
## 用户修改意见
## Agent 失败点
## 可复用步骤
## 不值得 Skill 化的部分
```

## 调研简报模板

```markdown
# Skill 调研简报

## 用户与场景
## 高频 query
## 输入材料
## 期望输出
## 当前流程
## 专家判断
## 可复用资产
## 失败案例
## 竞品 / 参考 Skill
## 初步机会评分
```

## 最低调研标准

- 5 条真实或高保真用户 query。
- 3 个成功案例或可接受的成功样例。
- 3 个失败案例或高保真失败模拟。
- 能说清无 Skill 时 Agent 会在哪里漏步骤、误判或产出不稳定。

## 基线失败记录

写 Skill 前先记录无 Skill 基线。推荐格式：

```markdown
# Baseline Failure

## 场景
## 用户请求
## 无 Skill 行为
## 具体失败
## 失败原因
## Skill 应改变什么行为
```
