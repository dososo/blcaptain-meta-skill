# Skill 设计与打包

## 加载时机

当需要设计 `SKILL.md`、`description`、目录结构、资源边界、模板或平台包时加载。

## 文件边界

`SKILL.md` 只放：

- 核心原则。
- 工作流程。
- 资源导航。
- 强约束。
- gotchas。
- 验证步骤。

`references/` 放长知识、案例库、API、流程细节、平台差异。  
`scripts/` 放确定性校验、转换、导出、解析、批处理。  
`assets/` 放模板、主题、schema、样例输入输出、静态素材。  
`evals/` 放路由、执行、压力、回归和失败库数据。  
`manifest.json` 放治理元数据，不塞进入口正文。

## description 写法

目标是路由，不是宣传。

```yaml
description: Use when [具体任务场景] for [输入/输出/对象], especially when [典型触发条件].
```

要求：

- 以触发条件为主。
- 覆盖用户真实说法和关键同义词。
- 不写“超强”“高级”“全能”之类营销词。
- 不在 description 里总结完整流程，避免 Agent 只看描述就跳过正文。
- 修改 description 后重跑正例、反例、近邻例。

## 设计方案模板

```markdown
# Skill Design Spec

## 名称
## description
## 目标用户
## 适用场景
## 不适用场景
## 输入
## 输出
## 核心流程
## 文件结构
## references
## scripts
## assets
## eval 设计
## gotchas
## 安全边界
## 成熟度层级
## 发布计划
```

## 开发顺序

1. 写 eval 场景草案。
2. 写 description。
3. 写最小 `SKILL.md`。
4. 补 references、scripts、assets。
5. 根据测试补 gotchas。
6. 删除不改变 Agent 行为的内容。

## 资源验收

- 每个资源文件都被 `SKILL.md` 直接引用。
- 每个资源文件都有明确加载条件。
- 每个脚本都能独立运行。
- 每个模板都对应一个输出场景。
- 没有空目录、占位文件、未来可能用到的装饰资产。

## 品味转约束

不要写“要高级”“要专业”。改成：

| 专家表达 | 可执行约束 |
| --- | --- |
| 不要廉价感 | 禁止无意义渐变、发光、过度圆角和堆 emoji |
| 文字别压脸 | 标题不得覆盖五官、关键 UI 或核心数据 |
| 别像模板套出来 | 连续页面不得同节奏、同构图 |
| 不要乱给法律意见 | 高风险结论必须标记人工复核 |
