# 多角色 Playbook

## 加载时机

当用户要求面向产品、运营、开发、测试、设计师、工程师、创作者或行业专家协作交付 Skill 时加载。

## 角色分工

| 角色 | 核心职责 | 交付物 |
| --- | --- | --- |
| AI 用户 | 提供真实任务、反馈、偏好和失败样本 | 原始 query、修改意见、成功标准 |
| 产品 | 定义目标用户、边界、体验和价值 | Skill Brief、NABC、发布标准 |
| 运营 | 分发、案例、反馈闭环 | 发布页、案例、用户反馈 |
| 开发 / 工程 | 文件结构、脚本、工具、验证命令 | Skill 包、脚本、依赖说明 |
| 测试 | 触发、反触发、执行、回归 | eval、失败库、测试报告 |
| 设计师 | 品味、版式、品牌和视觉禁区 | 模板、主题、设计 gotchas |
| 创作者 | 内容飞轮、案例包装、传播 | 示例输入输出、内容素材 |
| 行业专家 | 专业判断、禁区、复核标准 | SOP、人工升级规则 |

## RACI 简版

| 工作项 | 负责人 | 参与者 | 咨询者 |
| --- | --- | --- | --- |
| 需求定义 | 产品 | 用户、运营 | 行业专家 |
| 专家经验抽取 | 行业专家 | 产品、创作者 | 测试 |
| Skill 架构 | 工程师 | 产品 | 测试 |
| eval 设计 | 测试 | 开发、产品 | 行业专家 |
| 发布运营 | 运营 | 创作者、产品 | 开发 |
| 迭代决策 | 产品 | 全角色 | 用户 |

## 交接规则

- 产品给工程：用户、场景、输入输出、边界。
- 专家给产品：判断标准、禁忌、失败案例。
- 设计给工程：模板、主题、视觉约束、检查标准。
- 工程给测试：文件结构、脚本、依赖、运行方式。
- 测试给全员：eval、反例、失败报告。
- 运营给产品：案例、评论、传播数据、用户语言。

## 协作陷阱

- 产品只写愿景，不给真实 query。
- 专家只说“专业一点”，不解释什么算错。
- 工程先堆目录和脚本，没有 eval。
- 测试只测正例，不测反例和近邻。
- 运营只看传播数据，不看真实使用失败。
