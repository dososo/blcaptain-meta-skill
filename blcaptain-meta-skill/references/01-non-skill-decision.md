# Non-Skill 决策

## 加载时机

当需要判断某个需求是否值得做成 Skill、是否应降级为文档/脚本/项目规则/记忆时加载。

## 决策树

```text
用户需求
├─ 只需要解释、总结、翻译、一次性回答？ → 直接回答
├─ 只需要一份可复用文字材料，但不需要 Agent 执行？ → 写文档 / 笔记 / SOP
├─ 任务主要是确定性处理，路由不是难点？ → 写脚本 / CLI / 自动化工具
├─ 是项目内约定、命令、测试方式或协作规则？ → 写 AGENTS.md / CLAUDE.md / 项目说明
├─ 是个人偏好、长期习惯或风格记忆？ → 写 memory / 用户偏好
└─ 高频复用、需要路由、边界、流程、资源和验证？ → 才做 Skill
```

## 机会评分

每项 0-3 分，总分 24 分：

| 维度 | 判断问题 |
| --- | --- |
| 重复频次 | 是否经常重复，而不是一次性任务 |
| 单次价值 | 是否明显节省时间、提高质量或影响收入 |
| 失败成本 | 做错是否常见且代价明显 |
| 专家经验密度 | 是否依赖隐性判断、品味或行业边界 |
| 工具/资产复用 | 是否有脚本、模板、素材、schema 或案例可复用 |
| 可验证性 | 结果好坏是否能被检查或测试 |
| 分发潜力 | 是否有个人、团队或社区复用可能 |
| 维护可控性 | 是否能持续维护，不会变成大而全负担 |

建议：

- 0-8 分：不要做 Skill。
- 9-15 分：轻量 Skill、项目规则或文档。
- 16-20 分：正式 Skill。
- 21-24 分：产品化运营。

## 成熟度分层

| 层级 | 适用场景 | 默认产物 |
| --- | --- | --- |
| Scaffold | 探索期、个人使用 | 精简 `SKILL.md`，必要 references |
| Production | 团队复用、质量敏感 | 元数据、必要 references/scripts、基础 eval |
| Library | 组织级复用、跨环境维护 | 正反例、near-neighbor、维护元数据、回归证据 |
| Governed | 关键业务、meta skill、基础设施 | owner、review cadence、regression history、promotion evidence |

## 门禁选择

- 路由可能混淆：加 trigger gate。
- 多客户端分发：加 packaging gate。
- 团队复用：加 governance gate。
- 非琐碎 Skill：加 context gate。
- 新门禁只增加复杂度、不提升可靠性：删除或降级。

## 结论表达

每次判断都要输出：

```markdown
## 决策
做 / 不做 Skill

## 原因
## 替代方案
## 成熟度层级
## 最小可行交付物
## 不做的部分
```
